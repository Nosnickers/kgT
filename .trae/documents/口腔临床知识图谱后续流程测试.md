# 口腔临床知识图谱后续流程测试计划

## 目标
测试build-index、query、交互流程对口腔临床数据的适配性

## 测试步骤

### 1. 测试向量索引构建（build-index）
**目的**：验证能否正确处理口腔临床数据并生成向量索引

**操作**：
```bash
python qa_cli.py build-index --config .env --csv-file kg_build_20260112_225235.csv --output oral_kg_index.json
```

**验证点**：
- ✅ 成功读取CSV文件中的口腔临床实体和关系
- ✅ 成功生成实体和关系的嵌入向量
- ✅ 成功初始化向量存储
- ✅ 成功添加525个实体和1167个关系到向量存储
- ✅ 成功持久化向量数据
- ✅ 检查生成的索引文件包含正确的口腔临床数据

### 2. 测试查询功能（query）
**目的**：验证能否正确检索口腔临床实体和关系

**操作**：
```bash
python qa_cli.py interactive --config .env --vector-db chroma_db --retrieval-mode hybrid --top-k 5 --min-similarity 0.7
```

**测试问题**：
- "林某的患者有哪些就诊记录？"
- "牙齿16有什么诊断？"
- "Beyond冷光美白术使用了什么材料？"
- "张某的26号牙齿有什么症状？"

**验证点**：
- ✅ 成功初始化所有组件（embedding_manager、vector_store、retriever、qa_engine）
- ✅ 成功检索到相关的口腔临床实体和关系
- ✅ 成功生成包含检索知识的提示词
- ✅ LLM能够基于口腔临床知识生成准确的答案
- ✅ 答案中包含正确的FDI牙位编号（如16、26、36）
- ✅ 答案中包含正确的ICD-10诊断编码（如K02.1、K04.01）

### 3. 测试Web交互流程
**目的**：验证Web API接口是否正常工作

**操作**：
```bash
python web_server.py
```

**测试接口**：
- 健康检查：`GET http://localhost:5000/api/health`
- 统计信息：`GET http://localhost:5000/api/stats`
- 查询接口：`POST http://localhost:5000/api/query`（带JSON body）

**验证点**：
- ✅ Web服务器成功启动
- ✅ 所有组件初始化成功
- ✅ 健康检查返回正确的组件状态
- ✅ 统计接口返回正确的实体和关系数量（525个实体、1167个关系）
- ✅ 查询接口能够正确处理口腔临床问题
- ✅ 返回的答案准确且基于口腔临床知识

### 4. 验证生成的知识图谱数据
**目的**：确认Neo4j中的数据正确

**操作**：
```bash
python main.py --summary --config .env
```

**验证点**：
- ✅ 实体类型分布正确（牙齿225个、患者137个等）
- ✅ 关系类型分布正确（针对200个、属于136个等）
- ✅ 能够查询到具体的患者、牙齿、诊断、治疗信息

## 预期结果

所有流程应该能够：
1. ✅ 正确处理口腔临床数据（患者、牙齿、诊断、治疗等）
